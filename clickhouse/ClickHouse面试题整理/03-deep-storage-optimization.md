# ClickHouse 存储层极致优化与自研索引 (底层开发级)

## 1. 深度解析：ClickHouse 的数据压缩算法 (LZ4, ZSTD) 在不同场景下的选型逻辑
**标准答案：**
1. **LZ4 (默认)**：
   - **原理**：注重压缩/解压速度，CPU 开销极低。
   - **场景**：适用于 IO 瓶颈不明显，但对查询延迟极其敏感的场景。
2. **ZSTD**：
   - **原理**：提供更高的压缩比，但解压时 CPU 消耗较大。
   - **场景**：适用于冷数据存储、历史账单数据，或者磁盘 IO 是核心瓶颈的场景。
3. **列级压缩配置**：
   - 在 60k 级别的方案中，应提到“混合存储策略”：对频繁查询的维度列使用 LZ4，对大文本或稀疏指标列使用 ZSTD。
   - 甚至可以利用 `Delta` 或 `DoubleDelta` 编码（针对时间序列数据）进一步提升压缩效率。

## 2. 深入底层：ClickHouse 的 Wide Part vs Compact Part 存储格式
**标准答案：**
这是 `MergeTree` 引擎在处理不同规模数据时的自我优化：
1. **Compact Part (紧凑格式)**：
   - **结构**：所有列的数据都存储在一个 `.bin` 文件中。
   - **目的**：减少小批量写入时产生的文件句柄数量，降低文件系统压力。
   - **触发**：由 `min_rows_for_wide_part` 和 `min_bytes_for_wide_part` 控制。
2. **Wide Part (宽格式)**：
   - **结构**：每一列都有独立的 `.bin` 和 `.mrk` 文件。
   - **目的**：实现真正的列式并行读取，这是 ClickHouse 大规模查询性能的核心。
3. **面试深度**：解释在数据合并过程中，Part 是如何从 Compact 转换为 Wide 的，以及这种转换对 IO 模式的影响。

## 3. 专家级考点：ClickHouse 如何处理存储分层 (Tiered Storage)？
**标准答案：**
在大厂超大规模集群中，不可能所有数据都放在 SSD 上：
1. **存储策略 (Storage Policy)**：通过配置 `disks` 和 `policies`，定义热数据（SSD）、温数据（HDD）和冷数据（S3/HDFS）。
2. **自动迁移**：
   - `move_factor`：当 SSD 空间不足时，自动将旧 Part 移动到 HDD。
   - `TTL` 触发：在 SQL 层面定义 `TTL ... TO DISK 'hdd'` 或 `TTL ... TO VOLUME 'cold'`。
3. **零拷贝迁移**：解释 ClickHouse 如何在不影响查询的情况下，在底层文件系统层面完成 Part 的物理移动。

## 4. 自研索引思想：如果让你设计一个针对地理空间或全文检索的 Skip Index，你会怎么做？
**标准答案：**
这是一个考察架构设计能力的开放题：
1. **地理空间 (Geo)**：
   - 方案：引入 `S2 几何索引` 或 `H3 网格索引`。
   - 实现：在 Skip Index 中存储每个 Granule 的 H3 Cell ID 范围，查询时利用位运算快速过滤。
2. **全文检索 (Full-text)**：
   - 方案：自研 `Inverted Index` (倒排索引)。
   - 实现：ClickHouse 近期版本已引入实验性的倒排索引。核心在于如何将倒排表与稀疏索引的 Mark 机制结合，避免像 Elasticsearch 那样产生巨大的分词开销。
3. **核心思想**：所有的二级索引本质上都是在“空间”和“时间”之间做权衡，目的是尽可能多地跳过不相关的 Granule。
