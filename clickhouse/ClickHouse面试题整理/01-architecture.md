# ClickHouse 核心架构面试题 (中级/高级)

## 1. ClickHouse 为什么查询速度这么快？
**标准答案：**
ClickHouse 的高性能源于其针对 OLAP 场景的极致设计，主要包括以下几个核心点：
1. **列式存储 (Column-Oriented Storage)**：
   - 最小化 IO：只读取查询所需的列，减少数据扫描量。
   - 高压缩比：同列数据类型一致，压缩效果极佳（如 LZ4, ZSTD），进一步减少存储和 IO 开销。
2. **向量化执行引擎 (Vectorized Execution Engine)**：
   - 利用 CPU 的 SIMD (Single Instruction, Multiple Data) 指令集，一次处理一批数据（Vector），而不是逐行处理。
   - 极大地降低了函数调用和分支预测的开销。
3. **多核并行处理**：
   - ClickHouse 会利用服务器上所有的 CPU 核心来并行处理单个查询。
4. **索引设计**：
   - 稀疏索引 (Sparse Index)：配合数据排序，极大减少了索引占用空间，使其能常驻内存。
5. **硬件性能的极致利用**：
   - 针对机械硬盘（顺序读写优化）和 SSD 都有针对性的设计。
   - 尽量避免随机读写。

## 2. ClickHouse 的数据模型和表引擎体系是怎样的？
**标准答案：**
ClickHouse 的表引擎决定了数据的存储方式、索引方式以及支持的查询特性。
1. **MergeTree 系列 (核心)**：
   - `MergeTree`：最基础、最强大的引擎，支持分区、索引、数据压缩。
   - `ReplacingMergeTree`：支持数据去重（在后台合并时）。
   - `SummingMergeTree`：支持预聚合。
   - `AggregatingMergeTree`：支持更复杂的聚合状态存储。
   - `VersionedCollapsingMergeTree`：支持通过版本号进行行折叠（用于更新/删除模拟）。
2. **Log 系列**：
   - `TinyLog`, `Log`, `StripeLog`：适用于超小表或临时数据，不支持索引。
3. **集成引擎**：
   - `Kafka`, `MySQL`, `PostgreSQL`, `HDFS`, `S3`：用于与其他系统集成，直接读取外部数据。
4. **特殊引擎**：
   - `Distributed`：分布式表，本身不存储数据，用于将查询分发到多个分片。
   - `View`, `MaterializedView`：视图和物化视图。

## 3. 什么是 ClickHouse 的稀疏索引？它与传统数据库的 B+ 树索引有什么区别？
**标准答案：**
1. **稀疏索引 (Sparse Index)**：
   - ClickHouse 的 `MergeTree` 默认创建稀疏索引。它并不为每一行数据创建一个索引项，而是按照 `index_granularity`（默认 8192 行）为一个块（Mark）创建一个索引项。
   - 索引项记录的是该块中排序列的最小值。
2. **区别**：
   - **数据密度**：B+ 树是稠密索引（通常为每一行或每一页建立索引），而 ClickHouse 是稀疏索引。
   - **内存占用**：稀疏索引非常小，可以完全装载在内存中。
   - **定位方式**：B+ 树可以直接定位到行；稀疏索引只能定位到“数据范围”（Granule），然后进行顺序扫描。
   - **适用场景**：B+ 树适合 OLTP 的点查询；稀疏索引配合列存，适合 OLAP 的大批量范围扫描。

## 4. ClickHouse 的分区 (Partition) 和分片 (Shard) 有什么区别？
**标准答案：**
1. **分区 (Partition)**：
   - **逻辑单位**：在单个节点内部，将表数据按照某个键（通常是日期）拆分成不同的文件夹。
   - **目的**：查询优化（分区剪枝），方便数据管理（按天删除数据）。
   - **实现**：在 `MergeTree` 引擎中使用 `PARTITION BY` 定义。
2. **分片 (Shard)**：
   - **物理单位**：将数据水平切分到不同的物理节点/服务器上。
   - **目的**：横向扩展（Scale-out），利用多机性能处理超大规模数据。
   - **实现**：通过 `Distributed` 引擎和集群配置实现。

## 5. ClickHouse 是否支持事务？
**标准答案：**
ClickHouse 对事务的支持非常有限：
1. **不支持 ACID 事务**：不支持跨表的原子性更新或回滚。
2. **单表原子性**：在单个 `INSERT` 语句中，如果写入同一个分区，通常是原子的。
3. **最终一致性**：在分布式环境下，ClickHouse 追求的是最终一致性，而不是强一致性。
4. **应用场景**：ClickHouse 设计初衷是高吞吐写入和查询，事务开销会严重拖慢性能。如果需要事务，通常在下游（如 MySQL/PostgreSQL）处理好，再批量导入 ClickHouse。
